# FFAlgNodePolyDiv design note

This note records the Stage 0 reconnaissance for a future Mathematica command:

```wolfram
FFAlgNodePolyDiv[graphName, outputName, inputNodes,
  {targetTakePattern, idealTakePattern}, variables]
```

Only degree reverse lexicographic order is in scope for the first version.

## Existing FiniteFlow32 API pattern

1. Existing `AlgNode`-style Mathematica commands are implemented in three layers.  `mathlink/FiniteFlow32.m` validates Mathematica arguments and calls `FFRegisterAlgorithm`.  `mathlink/fflowmlink.cc` exposes a `LibraryFunctionLoad` target that translates Wolfram LibraryLink packets into C++ data.  The C++ algorithm class lives under `include/fflow` and `src`, and is inserted into a `Graph` with `Graph::new_node`.

2. `FFSparseSolverLearn` calls `FFLearnImplem[GetGraphId[gid]]`, checks `FFSolverIsImpossibleImplem`, and formats the native learned data as Mathematica rules.  The native learning path is `Session::learn`, `Graph::learn_all`, and each algorithm's `learn` method.

3. Node output sizes after learning are stored in `Algorithm::nparsout`.  For algorithms with `min_learn_times() == 0`, `Graph::new_node` marks the node learned and sizes `GraphData::xout_` immediately.  For learning algorithms, `Session::learn` calls `graph.learn_all(...)` and then sizes the output buffer from the learned `nparsout`.

4. Input nodes are referenced as native node ids.  Mathematica maps user symbols to native ids with `FFAlgId[{graph,node}]`; `FFRegisterAlgorithm` turns the user-provided `inputNodes` list into those ids.  Native evaluation receives each input node's evaluated output as `AlgInput xin[]`, where `xin[j][k]` is entry `k` of input node `j`.

5. Existing Mathematica take patterns are converted before entering C++.  For example, `TakeElemsToInternal` validates one-based `{nodeIndex, outputIndex}` lists, converts them to zero-based integers, and passes compact integer arrays to LibraryLink.  `FFAlgNodePolyDiv` should follow that pattern: parse monomials on the Mathematica side into exponent vectors, validate node/output indices, and pass compact integer data to C++.

6. The current finite-field prime inside a node is available through the `Mod mod` argument passed to `Algorithm::evaluate` and `Algorithm::learn`.  The active prime is `mod.n()`.  The msolve prime must be exactly this value.

## Prime table change

7. The FiniteFlow32 32-bit prime table is declared in `include/fflow/primes32.hh`, included into `src/primes.cc`, and stored as generated values in `src/ff_prime32_list.hh`.

8. The table should be regenerated by `scripts/generate_primes32.py`.  For msolve compatibility, the generator now uses `LIMIT = 2**31` and emits the 400 largest primes strictly below `2^31`.

## msolve integration plan

9. msolve should be called through an explicitly resolved executable path.  The first version can use `MSOLVE=/path/to/msolve` as an override, otherwise search `PATH`.  If not found, fail clearly.

10. The first implementation should shell out to the `msolve` executable.  Direct linkage should wait until the command-line prototype, parser, and Mathematica integration are correct.

11. The local msolve executable is version `0.9.4`.  Its normal-form syntax is:

```bash
msolve -f input.ms -o output.ms -n NF -v 0
```

`-n NF` computes normal forms of the last `NF` input polynomials with respect to a degree reverse lexicographic Groebner basis of the first `n - NF` input polynomials.  The help also mentions combining `-n` with `-i` to assert that the first polynomials are already a grevlex Groebner basis, but this option is not listed in the local help output and should not be used for Version A.

12. The msolve input format is:

```text
x,y,z
p
ideal_1,
ideal_2,
target_1,
target_2
```

The first line is a comma-separated variable list.  The second line is the prime field characteristic.  Each following line is one polynomial; commas separate polynomials, with no comma after the last polynomial.

13. For normal forms, the observed msolve output is a bracketed comma-separated list of reduced polynomials ending in `:`.  A local check with two targets produced:

```text
[1*y^1+1,
0]:
```

The parser should accept this list format, split top-level polynomial entries, and parse terms such as `c*x^a*y^b`, constants, and zero.

14. Temporary files should be created in a unique directory under the system temp directory.  Successful runs should clean up.  Failed runs should either leave a documented debug directory or copy input/output/stderr into a configured debug directory before returning a clear error.

## Proposed staged implementation

- Version A: shell out once per learn/evaluate call, with clear errors and deterministic temporary files.
- Version B: cache variable names, parsed take patterns, exponent vectors, learned monomial basis, and msolve path.
- Version C: investigate lower-overhead execution or direct linkage only after correctness tests are stable.

## Implemented Stages 2-6

The reusable native prototype layer lives in `include/fflow/poly_reduction.hh`
and `src/poly_reduction.cc`.

- Stage 2: `msolve_normal_forms_text` accepts a prime, variables, ideal
  generators, and target polynomials as text; writes an msolve input file; runs
  `msolve -n`; parses the bracketed normal-form output; and reports clear
  errors for missing msolve, unsupported primes, failed execution, and malformed
  output.
- Stage 3: `SparsePolynomial`, `PolyMonomial`, and `PolyTerm` provide sparse
  polynomial storage with coefficients modulo the active prime, exponent-vector
  monomials, deterministic degree reverse lexicographic sorting, msolve input
  rendering, msolve output parsing, like-term collection, and zero-term removal.
- Stage 4: `PolyTakePattern` and helpers convert compact flattened
  source/exponent arrays into take patterns, validate input-node and
  output-index bounds, and expand evaluated input-node data into concrete sparse
  polynomials.
- Stage 5: `learn_poly_reduction_basis` accepts evaluated input slices,
  expands the target and ideal take patterns for each slice, calls msolve
  normal-form mode, collects the monomials appearing in the reduced targets,
  sorts the learned basis in descending degree reverse lexicographic order so
  the highest-weight monomial is leftmost and the lowest-weight monomial is
  rightmost, reports the flattened runtime output size, and fails clearly when
  strict learning sees an inconsistent monomial basis across slices.
- Stage 6: `PolyDiv` is a native FiniteFlow32 algorithm node exposed as
  `FFAlgNodePolyDiv` through the C API, LibraryLink, and Mathematica package.
  Its learning phase stores the grevlex monomial basis, `FFPolyDivLearn`
  formats that basis as Mathematica monomials, and runtime evaluation reduces
  targets with msolve and returns the flattened coefficient matrix in the
  learned basis order.  Runtime normal forms containing a monomial outside the
  learned basis fail with a clear error.

The native test target `testpoly_reduction`, run through
`scripts/test_poly_reduction.sh`, covers these stages independently of the
FiniteFlow graph system.

The Mathematica integration test `tests/mathematica/poly_div.m`, run through
`scripts/test_mathematica_poly_div.sh`, covers graph construction, learning,
and runtime evaluation through the public Mathematica API.

## High-level Mathematica wrapper

`FFAlgPolyDiv[graph, node, targets, ideal, variables, parameters, opts]` is a
Mathematica-only convenience wrapper around `FFAlgNodePolyDiv`.  It parses
symbolic target and ideal polynomials in the supplied polynomial variables,
extracts coefficient expressions, validates that every non-variable coefficient
symbol is present in the explicit `parameters` list, creates the internal
coefficient-evaluation node with
`FFAlgRatFunEval` (or `FFAlgRatNumEval` when there are no parameters), builds
the take patterns, and then calls the lower-level node.

The parameter list is intentionally explicit because `FFAlgRatFunEval` maps
numerical graph-input entries by position.  Inferring parameters from the
coefficient expressions can silently swap values when the graph input node uses
a different order from the expression traversal order.

The wrapper supports only `MonomialOrder -> "DegreeReverseLexicographic"`.
It does not run learning, graph evaluation, reconstruction, or msolve directly.
The Mathematica test
`tests/mathematica/poly_div_wrapper.m`, run through
`scripts/test_mathematica_poly_div_wrapper.sh`, covers explicit parameter
ordering, rational coefficients, unsupported options, unsupported orders,
invalid polynomials, invalid variable lists, invalid parameter lists,
non-rational coefficient expressions, and the no-learning side-effect.

## LeadingMonomials helper

`LeadingMonomials[ideal, variables]` is a Mathematica-level helper for
inspecting the degree-reverse-lexicographic leading ideal of a symbolic ideal.
It does not implement Buchberger/F4 in Mathematica.  Instead it specializes
all symbols in `ideal` that are not listed in `variables` to random nonzero
integers below `2^31`, writes an msolve prime-field input file, and calls:

```bash
msolve -f input.ms -o output.ms -g 1 -v 0
```

Local msolve 0.9.4 documents `-g 1` as printing the leading ideal of the
grevlex reduced Groebner basis.  msolve prints those monomials in increasing
leading-monomial order; the Mathematica helper re-sorts them into the
FiniteFlow32 polynomial-division convention, with the highest-weight grevlex
monomial first and the lowest-weight monomial last.  The helper accepts
`"RandomSeed" -> seed` for deterministic parameter sampling in tests,
`"Retries" -> n` for retrying unlucky specializations, `"Prime" -> p` for an
explicit msolve-compatible prime, and `"MsolveExecutable" -> path` for tests
or nonstandard installs.
